## 2024-04-27 - Fix DoS via Atom Exhaustion in OrchestratorWorker
**Vulnerability:** User-provided inputs (`env` and `task` arguments in Oban jobs) were being converted to atoms dynamically using `String.to_atom/1` in `lib/lang/workers/orchestrator_worker.ex`. Because the Erlang VM does not garbage collect the atom table (with a limit around 1,048,576), an attacker could exhaust the atom table by providing arbitrary or randomized strings, causing the entire VM to crash (Denial of Service).
**Learning:** `String.to_atom/1` should never be used on untrusted input, particularly inputs coming from external sources or dynamically generated arguments for background jobs like Oban.
**Prevention:** Always use `String.to_existing_atom/1` for dynamically generated atoms that must match predefined system atoms, wrapping it in an isolated `try/rescue` block targeting `ArgumentError` to safely fail on unknown strings without exposing internal application errors. Pattern-match directly on strings for control flow (`case string`) when possible.
