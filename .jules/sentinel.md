## 2025-05-24 - [CRITICAL] Denial of Service via Atom Exhaustion in Oban Workers
**Vulnerability:** The application converts unvalidated Oban job arguments directly to atoms using `String.to_atom/1` in multiple worker modules (e.g., `ProductivityMetricsWorker`). Since the Erlang VM has a hard limit on the number of unique atoms it can store (~1 million), and atoms are never garbage collected, a malicious user or rogue process could exhaust the atom table by submitting jobs with dynamically generated, unique argument values (like random strings for `period_type`). This leads to an unrecoverable VM crash and a Denial of Service (DoS).
**Learning:** Oban job arguments originate from the database, meaning they are a form of external input when re-processing or when created from API endpoints. Converting any dynamic string from a database or JSON payload directly to an atom is inherently dangerous in Elixir/Erlang.
**Prevention:** Always use `String.to_existing_atom/1` when parsing external data into atoms. Wrap the conversion in a `try/rescue` block to catch `ArgumentError`, log a security warning if needed, and gracefully fall back to a safe default or reject the payload.
