## 2024-05-24 - DoS via Atom Exhaustion in Analytics Workers
**Vulnerability:** Use of `String.to_atom/1` to dynamically convert user-supplied or untrusted input (specifically the `"period_type"` from Oban job `args` which are deserialized JSON) into Erlang atoms.
**Learning:** Because Erlang atoms are not garbage collected, dynamically creating atoms from external/untrusted input allows an attacker or malformed data to exhaust the atom table (default limit is 1,048,576), crashing the Erlang VM and causing a Denial of Service (DoS). This pattern is especially problematic in background workers (like Oban jobs) where data traverses serialization boundaries.
**Prevention:** Always use `String.to_existing_atom/1` for converting strings to atoms when the input comes from an external source or serialized payload like an Oban job argument. Wrap the conversion in a narrow `try/rescue` block to catch `ArgumentError`, log a security warning, and gracefully fall back to a safe default (e.g., `:daily`) or fail the job explicitly.
