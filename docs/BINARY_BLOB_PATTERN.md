# Binary Blob Anti-Pattern in Version Control

## What Happened in This Repository

This repository accidentally tracked several categories of files that should
**never** live in version control:

| Pattern | Files found | Root cause |
|---|---|---|
| `_build/` | 6 BEAM files | No `.gitignore`; Elixir build output committed |
| `*.beam` (root-level) | 3 BEAM files | Loose compiled modules dropped in project root |
| `deps/` | 2,603 files (incl. `.so`, `.exe`) | Elixir dependencies committed instead of fetched |
| `erl_crash.dump` | 1 file (11.8 MB) | Erlang VM crash artifact committed |
| `.DS_Store` | 25 files | macOS Finder metadata auto-created and committed |
| `tmp/` | 4 log files | Debug logs committed |

These were all present because **the repository had no `.gitignore` file**.

---

## Why This Is Harmful

### 1. Binary blobs inflate repo size permanently

Git stores the full content of every committed file in its object database.
Deleting a file in a later commit does **not** reclaim that space — the binary
is forever baked into the history.  Cloning or fetching the repository forces
every contributor and every CI runner to download those bytes.

The `deps/jsonld_ex/priv/native/libjsonld_nif.so` alone was **883 KB** of a
Linux-specific shared library.  The `erl_crash.dump` was **11.8 MB**.

### 2. Architecture-pinning ("CPU pinning")

Compiled BEAM files and native shared objects (`.so`, `.exe`, `.nif`) are
platform-specific.  A BEAM file compiled on one OTP/Elixir version will be
rejected or silently mis-behave on a different version.  A `.so` compiled
for `x86_64-linux-gnu` is unusable on `arm64` (Apple Silicon, AWS Graviton).

When these files are committed:

- CI may load the committed artifact **instead of compiling from source**,
  silently running the wrong binary.
- Adding a new architecture to CI (e.g. moving to ARM runners) causes
  mysterious `cannot open shared object file` or `beam: error loading NIF`
  failures.
- Mix's cache-busting logic cannot distinguish a correct build from one
  produced on a different OS/Elixir version, so stale artifacts go undetected.

### 3. Automated flows stall ("halts our automated flows")

When `deps/` is committed, `mix deps.get` still runs but ends up diffing
2,000+ files against the checked-out tree to decide whether anything changed.
On a cold GitHub Actions runner this triggers:

1. A large checkout step (all the dep source + native binaries transferred).
2. `mix deps.get` re-checksums every dep file to confirm they match `mix.lock`.
3. Mix's dependency compilation step, which re-reads every committed dep to
   produce `_build/` — even when nothing changed.

The committed `_build/` BEAM files add a second round of disk I/O because Mix
validates their build metadata against the current Elixir/Erlang version on
every compile invocation, then re-emits them when they don't match.

Together this inflated typical CI compile times significantly and caused
non-deterministic failures when the committed binaries were stale relative to
the BEAM version on the runner.

---

## When Was It Introduced

The initial repository commit (`de88bf9`) had no `.gitignore`.  Because
Elixir's tooling writes `_build/` and `deps/` into the project root by
default, the very first `git add .` swept them in.  The macOS `.DS_Store`
files and `erl_crash.dump` followed the same path: no exclusion rule, so they
were picked up by subsequent broad `git add .` invocations.

The root-level `.beam` files (`Elixir.Lang.MCP.AdvancedProxy.beam`, etc.)
appear to have been generated by running `elixirc` or `mix compile` directly
in the project root, then added without noticing they were build artifacts.

---

## How It Was Fixed

1. A `.gitignore` was added covering all standard Elixir/Phoenix/macOS build
   artifacts (see `.gitignore` at the repository root).
2. All tracked artifacts were removed from the git index with
   `git rm --cached --recursive` — the files remain on disk for local
   development but are no longer tracked by git.

> **Note:** The blobs still exist in git *history*.  If the repo ever needs to
> be fully cleaned (e.g. to satisfy a security audit or to dramatically reduce
> clone size), use `git filter-repo --path <pattern> --invert-paths` or
> BFG Repo Cleaner.  That is a destructive history rewrite and must be
> coordinated with all contributors.

## Pattern Checklist — What to Look For in Other Projects

Use this checklist when auditing a repository for the same anti-pattern:

```bash
# 1. Does the repo have a .gitignore at all?
test -f .gitignore || echo "MISSING .gitignore"

# 2. Are build directories tracked?
git ls-files | grep -E "^(_build|dist|build|out|target)/" | head -5

# 3. Are dependency directories tracked?
git ls-files | grep -E "^(deps|node_modules|vendor|packages)/" | head -5

# 4. Are compiled binaries tracked?
git ls-files | grep -E "\.(beam|pyc|class|o|so|dylib|dll|exe|nif|native)$" | head -10

# 5. Are crash / debug artifacts tracked?
git ls-files | grep -E "(erl_crash\.dump|\.log|core\.[0-9]+|heapdump)$" | head -5

# 6. Are OS metadata files tracked?
git ls-files | grep -E "\.DS_Store$|Thumbs\.db$|desktop\.ini$" | head -5

# 7. How large are the tracked blobs?
git ls-files -z \
  | xargs -0 git cat-file --batch-check='%(objectsize) %(path)' \
  | sort -rn \
  | head -20
```

If any of the above return hits, follow the remediation steps below.

---

## Remediation Steps (Copy-Paste)

```bash
# Stop tracking the offending files without deleting them from disk
git rm -r --cached _build/ deps/ dist/ node_modules/ 2>/dev/null || true
git rm --cached $(git ls-files | grep -E "\.(beam|pyc|class|o|so|dylib|dll|exe|nif)$") 2>/dev/null || true
git rm --cached $(git ls-files | grep -E "erl_crash\.dump|\.DS_Store|Thumbs\.db") 2>/dev/null || true

# Create (or extend) .gitignore — use a generator such as
# https://www.toptal.com/developers/gitignore for your language stack
# then add the generated output to your .gitignore

# Commit the cleanup
git add .gitignore
git commit -m "chore: remove build artifacts and add .gitignore"
git push
```

## Language-Specific Defaults

### Elixir / Phoenix

```gitignore
_build/
deps/
*.beam
erl_crash.dump
tmp/
priv/static/assets/
assets/node_modules/
native/*/target/
.env
```

### Node / JavaScript

```gitignore
node_modules/
dist/
build/
.cache/
*.log
.env
```

### Python

```gitignore
__pycache__/
*.pyc
*.pyo
.venv/
venv/
dist/
build/
*.egg-info/
.env
```

### Rust

```gitignore
target/
Cargo.lock   # only for libraries; keep for binaries
*.rlib
```

### Java / Kotlin (Maven / Gradle)

```gitignore
target/
build/
*.class
*.jar
.gradle/
```
