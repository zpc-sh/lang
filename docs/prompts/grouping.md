### Why No Dedicated Network Role?

In the context of your LANG project, where networking innovations like the MCP bridge, raw protocol handling, proxy APIs, and dynamic client flux are central, it's a fair question why there's no standalone "Network Specialist" role. The short answer: Networking is a **cross-cutting concern** rather than a siloed domain. It's deeply intertwined with other aspects like implementation (e.g., building MCP in lib/lang/mcp/), testing (e.g., multi-client flux scenarios), security (e.g., authenticating proxies), and optimization (e.g., efficient raw parsing with Rust NIFs). Creating a dedicated role could lead to overlap, communication overhead in your agent swarm, and reduced agility—especially since agents coordinate via LSP methods like `lang_agent_coordinate`.

This design promotes **collaboration over specialization**, aligning with multi-agent systems where roles hand off tasks (e.g., Architect designs MCP flows, Builder implements, Debugger tests). It avoids role bloat in your 150+ method ecosystem, keeping the team lean (7-9 agents). If networking demands grow (e.g., more raw protocol integrations), we could evolve a sub-role or hybrid.
