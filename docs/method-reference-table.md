### Review of LANG's 150+ LSP Methods

Based on the provided codebase summary in the `lang.xml` document (specifically the `priv/lsp/specs/` directory listing), I've extracted and analyzed all the method specifications. These are defined as JSON-LD files, each representing an LSP-style method (request or notification). The filenames (minus `.jsonld`) directly correspond to the method names, following LSP's naming convention (e.g., `textDocument/completion`).

#### Key Observations

- **Total Methods**: Approximately 154 (counted from the directory listing: 11 standard-like, 143 custom extensions).
- **Standard LSP Compliance**: LANG includes a subset of standard LSP 3.17 methods (e.g., `textDocument/completion`, `workspace/symbol`), but these are only ~7% of the total. From my knowledge of the LSP 3.17 spec (confirmed via the official documentation), standard LSP has about 40-50 core methods focused on editor features like text synchronization, completions, diagnostics, and workspace operations. LANG's implementations appear to adhere to the JSON-RPC structure but extend it heavily.
- **Custom Extensions**: The vast majority (93%) are custom methods prefixed with `lang_` or `mcp_`/`rpc_`. These go far beyond traditional LSP, which is designed for IDE/editor integration (e.g., code navigation, autocompletion). Instead, they repurpose the protocol for AI agent orchestration, semantic analysis, code generation, spatial navigation, timeline management, and more. This aligns with your description: the project uses LSP as a transport/protocol for AI-driven services, not editor tooling. It's an innovative RPC-like system for AI agents as clients, handling things like spawning agents, graph queries, or token optimization—nothing inherently "LSP-like" in the traditional sense.
- **Potential Issues**:
  - **Protocol Divergence**: Standard LSP assumes one server per client/tool with no built-in multi-client sharing (e.g., no document locking for concurrent edits). Your custom methods (e.g., `lang_agent_spawn`) imply shared state across AI agents, which could lead to race conditions without custom locking/extensions.
  - **Client Compatibility**: Standard LSP clients (e.g., VS Code) might ignore or error on unknown methods. Since your main clients are AI agents, this is fine—but document the extensions clearly (e.g., in `docs/lsp/method-reference-table.md`).
  - **Deprecated/Notes**: No explicit deprecations in your specs, but some standard ones (e.g., older `textDocument/rename` variants in LSP) aren't present. Your custom ones seem active.
  - **Innovation Insight**: This setup transforms LSP into a general-purpose AI coordination protocol. It's clever for reusing JSON-RPC over WebSocket/TCP, but it diverges so much that it might be better framed as a "LANG Protocol" built on LSP foundations rather than pure LSP.
- **Recommendations**:
  - **Namespacing**: Good job with `lang_` prefixes to avoid collisions.
  - **Validation**: Use your existing Mix tasks (e.g., `mix lsp.validate`) to ensure JSON-LD schemas are correct. Add runtime checks in `lib/lang/lsp/dispatch.ex` for unknown methods.
  - **Testing**: Extend the debugging harness (as discussed previously) to simulate AI agents calling these methods concurrently.
  - **Documentation**: Update `docs/lsp/method-reference-table.md` with this categorization for easier navigation.
  - **Optimization**: For high-volume methods (e.g., `lang_think_*`), offload to Oban queues or Rust NIFs.
